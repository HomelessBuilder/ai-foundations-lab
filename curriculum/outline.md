## How to Read This Document

Each module below maps to exactly one lab. Every lab produces an artifact
the student keeps — this is the non-negotiable constraint from `VISION.md`
("Build, Don't Watch").

**Pacing:** Self-paced and flexible. Each lab is scoped to take roughly
60 minutes for the target student, but nothing is timed or gated by a
clock — a student can move faster or slower without penalty.

**Tips woven in, not lectured separately.** No module — including setup
— should feel like pure mechanics with the learning bolted on afterward.
Relevant knowledge gets attached directly to the action the student is
taking, in small doses, right when it's useful. This is a course-wide
principle, not just a Module 0 concern: baby-steps throughout, but never
steps that teach nothing.

For each module:
- **Objective** — what the student should understand by the end
- **Artifact** — the real thing they walk away with
- **Concepts introduced** — new ideas, kept to the minimum needed
- **Depends on** — what has to exist already for this lab to make sense

## Module 0 — Getting Set Up

**Lab 0: "Say Hello"**

- **Objective:** Student installs Claude Code and gets oriented to the
  course platform — and comes away with more than "I clicked the right
  buttons." No formal teaching happens here, but every step is a chance
  to plant a small, real seed of understanding in plain language, tied
  directly to the action the student is taking in that moment. The
  student should finish this module feeling like they accomplished
  something real, not just that they got through a checklist.
- **Artifact:** A saved transcript of the student's first real exchange
  with Claude Code — asking it to introduce itself, or explain what it
  just did. Small, but real: proof it works, and something Module 1 can
  point back to as a concrete example instead of an abstract one.
- **Concepts introduced:** Not a lecture — light, in-the-moment tips
  attached to the actions themselves. For example: a one-line note on
  what's actually being installed and why it lives outside the browser,
  offered right as the student installs it; a one-line note on what just
  happened being different from sending a chat message, offered right
  after the student's first exchange. These are seeds, not lessons —
  Module 1 is where they get named properly and built on — but the
  student shouldn't walk out of Module 0 having learned nothing.
- **Depends on:** Nothing. True day zero — before the student has built
  anything.

**Approved — additive, not merged.** Content depth revised per Diego's
note: setup carries real, if small, learning throughout — not a purely
mechanical checklist with concepts deferred entirely to Module 1.

## Module 1 — Understanding AI Tools

**Lab 1: "What Are We Actually Doing?"**

- **Objective:** Student understands what Claude Code is, how it differs
  from a chat window like ChatGPT, and what "context" means in practice —
  before touching any tools.
- **Artifact:** A short written reflection, saved as a plain text file:
  "What I Want to Build." A few sentences on what the student wants AI
  help with. Not precious — just something real and theirs.
- **Concepts introduced:** Chat vs. agent, what "context" means, why
  documentation matters, the core loop (context → build → document →
  re-enter) previewed at a high level.
- **Depends on:** Module 0's installed, working Claude Code session.
  Concepts are now taught by pointing at what the student already has
  open, rather than purely in the abstract.

**Approved.** The reflection artifact stands and feeds directly into Lab 2.

## Module 2 — Your First Project Folder

**Lab 2: "Give AI a Home"**

- **Objective:** Student sets up Claude Code, creates a project folder,
  and writes their first CLAUDE.md — their first experience of giving AI
  structured context instead of just chatting at it.
- **Artifact:** A real project folder with a CLAUDE.md the student wrote,
  seeded from their Lab 1 reflection.
- **Concepts introduced:** Folders as project boundaries, what belongs in
  a CLAUDE.md (vs. what belongs in conversation), the difference between
  telling AI something once and writing it down so it persists.
- **Depends on:** Lab 1's reflection file.

---

## Module 3 — The Learning Journal

**Lab 3: "Start the Paper Trail"**

- **Objective:** Student starts a personal learning journal and writes
  their first entry, understanding why re-entry context matters.
- **Artifact:** A journal folder with one real entry documenting what
  happened in Labs 1–2.
- **Concepts introduced:** Session documentation, why "future you" needs
  notes from "past you," first taste of the recursive principle — writing
  something down that will make the next session easier.
- **Depends on:** Project folder from Lab 2 (journal lives alongside it).

---

## Module 4 — Building Something Small

**Lab 4: "Build the Thing"**

- **Objective:** Student picks a real, small project and builds it with
  AI assistance — practicing iteration, course-correction, and asking
  better questions instead of getting it right on the first prompt.
- **Artifact:** A working micro-project, however small. A suggested project
  menu will accompany the Lab 4 content for students with no idea what to
  build, but a student's own idea is always welcome and encouraged.
- **Concepts introduced:** Iteration as normal (not failure), how to
  course-correct a prompt, the difference between a vague ask and a
  well-scoped one.
- **Depends on:** Project folder + CLAUDE.md from Lab 2.

---

## Module 5 — Documenting and Handing Off

**Lab 5: "Leave Yourself a Note"**

- **Objective:** Student logs the Lab 4 session in their journal, then
  experiences picking up where they left off using only their own
  documentation — the recursive payoff becomes real and visible here.
- **Artifact:** A second journal entry plus a handoff note, followed by
  a short exercise where the student re-enters the project cold and
  proves to themselves the notes worked.
- **Concepts introduced:** The handoff format, what makes a note useful
  to future-you vs. useless, session continuity without memory.
- **Depends on:** Journal from Lab 3, micro-project from Lab 4.

---

## Module 6 — Git as a Safety Net

**Lab 6: "Turn On the Undo Button"**

- **Objective:** Student learns just enough git — init, add, commit — to
  feel safe making changes. Explicitly not full version control theory.
- **Artifact:** Their project folder, now git-tracked, with a real commit
  history starting from Lab 4's micro-project.
- **Concepts introduced:** Git as an undo button (framing, not jargon),
  init/add/commit as a three-step habit, what a commit is in plain terms.
- **Depends on:** Micro-project from Lab 4.

---

## Module 7 — What Comes Next

**Lab 7: "Where This Goes"**

- **Objective:** Student sees how this foundation connects to more
  advanced usage and leaves with a concrete sense of their own next step
  — not just a course completion feeling.
- **Artifact:** A one-page "what's next" note, in the student's own words,
  listing 2–3 things they want to try next with what they've built. Filed
  in their journal as the final entry.
- **Concepts introduced:** None new — this module synthesizes rather than
  introduces. Light preview of where more advanced usage leads (without
  overwhelming).
- **Depends on:** Everything — this is the close.

**Approved.** The "what's next" note stands as the closing artifact.
